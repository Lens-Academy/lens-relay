# HTML Comments in Lens (v2)

**Date:** 2026-05-23

## Context

Builds on the v1 HTML sharing spec (`2026-05-22-html-sharing-v1-design.md`), which delivered Y.Text-backed `.html` editing with a sandboxed preview iframe. v1 explicitly deferred comments. v2 adds collaborative comments on `.html` documents that work in both the source view and the preview.

The core constraint: the v1 preview uses `<iframe sandbox="allow-scripts">` with no `allow-same-origin`. The parent cannot walk the iframe DOM directly. All preview-side comment behavior must flow through a `postMessage` bridge script that Lens injects into the iframe.

This spec covers v2 only: storage format, position-finding, bridge script, source/preview UI, creation flow. Out of scope (and listed at the end): comments on Markdown, suggestions/track-changes, @-mentions, notifications, comment search, reactions.

**Prerequisite:** v1 (`docs/superpowers/plans/2026-05-22-html-sharing-v1.md`) is implemented and shipped. Before drafting the implementation plan for v2, verify the v1 code matches the v1 spec — in particular the `HtmlPreview` srcdoc pipeline and the iframe sandbox tokens. If v1 diverged from spec, reconcile before planning.

## Data model

Comments are point-anchored, threaded, and stored as a series of HTML comment nodes embedded in `Y.Text('contents')`. Each comment node carries a single JSON payload.

### Two marker types

**Top-level comment** (one per thread; carries the point anchor in source):

```html
<!--lens-comment {"id":"c1","author":"luc@example.org","ts":"2026-05-23T14:33:00Z","body":"why does this loop run twice?"}-->
```

**Reply** (zero or more, immediately following the parent comment):

```html
<!--lens-reply {"id":"r1","parent":"c1","author":"jan@example.org","ts":"2026-05-23T14:35:00Z","body":"intentional, see comment in source"}-->
```

A thread = one `lens-comment` marker + zero or more `lens-reply` markers clustered immediately after it, in textual order = reply order.

### Encoding rules

- JSON inside HTML comments requires one escape: any `-->` appearing in string values is rewritten to `--&gt;` on write, reversed on read. No other escaping needed.
- IDs are UUIDv4. Author is the user's identity from the existing auth context. Timestamp is ISO 8601 (UTC).
- A `lens-comment` marker is always followed by:
  - Zero or more `lens-reply` markers whose `parent` matches its `id`, then
  - Any other content (text, tags, or a different `lens-comment` block starting a new cluster).
- The parser tolerates whitespace and newlines between markers within a cluster.
- A `lens-reply` whose `parent` doesn't match any preceding `lens-comment` in the same source is treated as orphaned data: logged, surfaced in the orphan panel, but otherwise ignored.

### Permissions

Anyone with edit access on the doc can create comments, add replies, and edit/delete their own messages. View-only users see comments but cannot mutate the doc. Comment-specific permissions (Google Docs-style "Comment only" tier) are deferred.

### Editing/deleting

A user can edit or delete only their own messages. Edit = atomic replace of the single marker (parse → modify payload → Yjs delete+insert in one transaction at `LENS_EDITOR_ORIGIN`). Delete = remove the marker. Deleting a parent comment cascades to its replies: the cluster is removed in a single transaction. There is no edit-window restriction.

### No resolve state in v2

Threads cannot be marked resolved. To remove a thread, delete the parent comment (which cascades to its replies). Resolve added in a later phase.

## Position-finding (preview click → source position)

When a user clicks somewhere in the preview to add a comment, Lens must determine the corresponding character position in `Y.Text('contents')`. The approach is two-tier.

### Tier 1: Fingerprint

The bridge captures the click target and reports a fingerprint:

- Up to 30 characters of `textContent` before the click point (within the click target's text node).
- Up to 30 characters after.
- The element's tag name plus attribute summary (`id`, first class token).
- An ancestor path: tag names plus `nth-of-type` indices walking up to `<body>`.
- The click rect in iframe coordinates.

Parent text-searches the source HTML for the before/after text snippet. Candidates are ranked by:

1. Ancestor-path match length (descending) — longer matched suffix beats shorter.
2. Text-context length matched (descending).
3. Source position (ascending) — earlier in doc as tiebreak.

Three outcomes:

- **Unique top candidate** with the next candidate substantially weaker (heuristic gap, e.g., 2+ ancestor levels) — place marker, done.
- **Ambiguous** (top candidate within the gap of others) — fall through to Tier 2.
- **No match** — text was generated by a script. Fall through to Tier 2 with an empty candidate list (forces split-source fallback).

### Tier 2: Probe-verify

For each candidate position (in ranked order, up to N=5 candidates):

1. Insert a unique probe marker into a copy of the source string: `<!--lens-probe TOKEN-->` where `TOKEN` is a random 16-byte hex string per placement gesture.
2. Render the modified source in a **hidden iframe** (offscreen via `position:fixed; left:-9999px; top:-9999px; width:1px; height:1px; visibility:hidden`, same sandbox as visible preview).
3. The bridge in the hidden iframe receives a `find-probe` RPC, scans for the probe comment node via TreeWalker, computes the bounding rect of its nearest rendered neighbor, reports back.
4. Parent compares: does the probe's rect overlap the original click rect within a 20px tolerance?
5. If yes: replace probe with the real comment marker in `Y.Text` (atomic transaction). Open thread popover.
6. If no: try the next candidate.

If all N candidates fail: surface a **split-source confirmation UI**. Open source view alongside preview; highlight the top candidate region in source; user clicks the correct spot. Manual confirmation always places the marker exactly where the user clicks in source.

The hidden iframe is lazily created on the first click-to-place gesture, reused across candidate attempts within that gesture, and torn down when the gesture completes or is cancelled. It does not persist across placement gestures (avoids stale-content drift).

## Bridge script

A small Lens-controlled JavaScript module injected at the top of every preview iframe (visible and hidden alike).

### Injection

Lens preprocesses the user's HTML before setting `srcdoc`:

1. If source contains `<head>` (case-insensitive), inject `<script>BRIDGE_SOURCE</script>` as the first child of `<head>`.
2. Otherwise, prepend `<script>BRIDGE_SOURCE</script>\n` to the source.

Browsers' tolerant HTML parsing accepts both forms. The injected script is a single build-time-bundled module, identical across docs, with a build-time hash used by the parent to confirm "ready" messages come from a script Lens shipped (and not a user-script masquerading).

### Postmessage protocol

All messages are JSON objects with `type`, `nonce`, and `payload` fields.

**Parent → bridge:**

| Type | Payload | Purpose |
|---|---|---|
| `init` | `{ nonce, comments: [{id, ...}] }` | Sent after iframe load. Provides the per-iframe nonce for message authentication; lists known comments so bridge can render dots. |
| `enable-click-to-place` | `{}` | Switch bridge into "next click is a placement gesture" mode. |
| `disable-click-to-place` | `{}` | Cancel placement mode. |
| `find-probe` | `{ token }` | (Hidden iframe only) Locate `<!--lens-probe TOKEN-->`, report rect of nearest rendered neighbor. |
| `highlight-comment` | `{ id }` | Pulse the dot for a given comment id (e.g., user clicked it in orphan panel). |
| `set-comments` | `{ comments: [...] }` | Update the rendered comment list (after add/edit/delete in source). |

**Bridge → parent:**

| Type | Payload | Purpose |
|---|---|---|
| `ready` | `{ scriptHash }` | Confirms bridge loaded; echoes its build hash. Parent verifies hash matches expected. |
| `click-captured` | `{ fingerprint }` | Click handler fired during placement mode; fingerprint as described above. |
| `dot-clicked` | `{ id }` | User clicked a comment dot overlay. |
| `probe-found` | `{ token, rect | null }` | Response to `find-probe`. `rect: null` if token not found. |
| `comments-rendered` | `{ found: [id], orphaned: [id] }` | After each rendering pass, reports which comment ids had/lacked DOM anchors. |

### Authentication

Each iframe gets a unique 16-byte hex nonce on init. Parent ignores any postMessage whose `event.source !== iframe.contentWindow` or whose `nonce` field doesn't match the one assigned to that iframe. Bridge stores the nonce from `init` and ignores any subsequent parent message whose nonce doesn't match.

The `ready` message from the bridge is a special case: it's sent before `init`, so it carries no nonce. Parent validates `event.source` and the `scriptHash` field on `ready`, then replies with `init` carrying the nonce.

### Rendering dots

On `init` and on every `MutationObserver` tick (debounced 100ms), the bridge:

1. Walks the DOM with `TreeWalker(NodeFilter.SHOW_COMMENT)` to find every `<!--lens-comment ...-->` node.
2. For each comment, finds the nearest rendered element: the next element sibling, or if none, the parent element.
3. Computes its bounding rect.
4. Creates or updates an absolutely-positioned overlay `<div class="lens-comment-dot">` styled as an inline 💬 indicator, anchored to that rect (top-right of element).
5. Attaches a click handler: clicking the dot sends `dot-clicked { id }` to parent.

If a `lens-comment` node has no findable rendered neighbor (entire body was rewritten by a script), it's omitted from rendered overlays and included in the `orphaned` list of the next `comments-rendered` message.

Overlay dots live in a Lens-controlled `<div id="lens-comment-overlay-root">` appended to `<body>` by the bridge. The root has `position: fixed; pointer-events: none; z-index: 2147483647;`; individual dots have `pointer-events: auto`. The root and its children are tagged with a `data-lens-overlay="true"` attribute, and the MutationObserver ignores changes within it.

## Parent-side UI

### Preview pane

- The existing v1 preview iframe, now with the injected bridge.
- A **floating popover** renders over the iframe when a dot is clicked or a new comment is being placed. Popover shows the thread (parent comment + replies in chronological order), allows reply input, and edit/delete on own messages. Closes on outside click or Esc. Popover position: anchored to the bridge-reported rect of the clicked dot, with collision-detection to keep it on screen.
- **Click-to-place:** user activates "comment mode" via a button in the editor chrome (next to the source/preview/split toggle). Activating sends `enable-click-to-place` to the bridge; cursor changes to a question/comment indicator inside the iframe (bridge applies a body-level class with a `cursor: ...` CSS rule). Next click in preview triggers the placement flow. On successful placement, popover opens with an empty body field for the first comment.

### Source pane (CodeMirror)

- Markers shown as raw HTML comment text. **No CodeMirror folding in v2.** The cluster pattern (parent + adjacent replies) is visible in source for direct inspection and manual editing.
- Editing a comment via the popover writes back to `Y.Text` atomically; CodeMirror reflects the change immediately via yCollab.
- A lightweight CodeMirror linter flags malformed marker payloads (invalid JSON, unknown fields, reply with no parent) with a margin marker. This is a v2 polish item; ship without if it slips.

### Orphaned comments

A small badge in the editor chrome shows the count of orphaned comments (from the bridge's `comments-rendered` reports). Click → opens a right-side panel listing each orphan with body, author, ts, and a "find in source" button that switches to source view scrolled to the marker.

## Server changes (`crates/relay/`)

None expected. Comments live in `Y.Text('contents')`, which already round-trips through the relay unchanged. The link indexer continues to skip `.html` files (per v1).

Verification step: confirm that `lens-comment` and `lens-reply` markers in source don't trip any existing server-side parser, webhook, or git-sync pipeline. Add a regression test that creates an `.html` doc with embedded comment markers via MCP and reads it back unchanged.

## Testing

**Comment store (pure unit):**

- Round-trip: parse markers from a source string, serialize back, verify byte-equal source.
- `-->` escape/unescape: comment bodies containing `-->` survive a round-trip.
- Cluster parsing: parent + replies in order; reply with non-matching `parent` reported as orphaned data.
- Malformed JSON in a marker → marker ignored, error logged, surrounding markers parse normally.
- Atomic edit: editing one comment in a doc with 10 comments mutates exactly one marker.
- Cascade delete: deleting a parent removes its replies in the same transaction.

**Position finder (pure unit):**

- Fingerprint scoring: unique-text fingerprint returns single high-confidence candidate.
- Ambiguous text returns ranked candidates; ranking matches the documented heuristic.
- No match returns empty list.
- Probe-verify with the hidden iframe mocked: candidate accepted on rect overlap, rejected on miss, falls through to next candidate.

**Bridge protocol (unit, no iframe):**

- Message round-trip with valid nonce → handled.
- Wrong nonce → ignored (no handler called).
- Wrong source → ignored.
- `ready` validates `scriptHash` against expected build hash.

**Bridge integration (component test in happy-dom):**

- Mount `HtmlPreview` with three sample comments → assert three dots rendered at expected positions (mock `getBoundingClientRect`).
- Simulate `dot-clicked` from bridge → assert popover opens with the matching thread.
- Click-to-place with a unique-text click → assert source receives a new `lens-comment` marker at the expected position; assert popover opens.
- Source has a script that wipes a comment node → assert it reports as orphaned in the panel.

**Manual / multi-tab:**

- Open the same doc in two browser tabs. Add a comment in tab A → verify it appears in tab B's preview (dot) and source (marker) within one debounce cycle.
- Two users reply simultaneously to the same thread → verify both replies persist as separate `lens-reply` markers; no JSON conflict; order = arrival order at the CRDT.
- Open an `.html` doc whose script calls `document.body.innerHTML = '...'` after load → verify comments are reported orphaned, still editable from source view.
- Open the `jsonl2html.py` output, add a comment on a turn → verify the dot lands at the right turn and the popover works.
- Permission check: open as a viewer (view-only token) → verify comment-mode toggle is disabled and existing dots are read-only.

## Key decisions (locked)

- **Marker form:** HTML comment nodes `<!--lens-comment ...-->` and `<!--lens-reply ...-->`.
- **Scope:** point-only.
- **Threading:** parent + flat list of replies (no nested).
- **Reply placement:** immediately after parent in source.
- **Encoding:** JSON inside the comment, `-->` escaped as `--&gt;`.
- **Permissions:** anyone with edit access.
- **Resolve state:** none in v2.
- **Position-finding:** fingerprint-first, probe-verify fallback, manual split-source last resort.
- **Probe-verify mechanism:** hidden iframe with same sandbox, re-render per candidate (up to 5).
- **Bridge:** injected at top of srcdoc; nonce-authenticated postMessage RPC; build-hash-verified at `ready`.
- **Preview UI:** inline 💬 dots at anchors + floating popover for threads.
- **Create flow:** comment-mode toggle + click in preview.
- **Source UI:** raw marker text in CodeMirror; orphans shown in sidebar.
